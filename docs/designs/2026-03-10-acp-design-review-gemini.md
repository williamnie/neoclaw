# ACP 多编程工具编排技术设计 Review (Gemini 视角)

日期：2026-03-10
审阅人：Gemini (Antigravity)

---

## 总体评价

在 Codex 提供的基础多节点架构和 Claude Code 给出的务实工程化（代码库集成、按阶段交付）建议之上，作为立足于大模型底层特性的 Agent，我需要指出方案中目前仍然缺失的**分布式系统强健性**和**模型实际协同瓶颈**问题。

现有的设计偏向于理想状态下的工具调用，却忽略了超长执行时间、幻觉放大、节点崩溃等实际工程难题。以下提出 5 个维度的深度重构建议。

---

## 一、架构上的致命缺陷：长时阻塞与大模型工具超时的矛盾

**问题描述：**
原设计中 `acp_workflow` 被设定为一个普通的同步工具调用（甚至等待验证结果）。但在实际场景中，一次“规划+开发”的工作流动辄耗时 10~30 分钟。任何主流大模型的 API 请求都会在几分钟内触发超时（Timeout），这会导致主 Agent 崩溃、会话上下文断裂。

**改进方案：**
* 修改工具语义，将 `acp_workflow` 更新为**“异步作业 (Async Job) 提交”**模式。
* 工具调用后立即返回 `job_id` 与确认信息（如：“已在后台启动开发流程”），释放大模型 API 的 HTTP 连接。
* 工作流在后台（Orchestrator）独立运行，运行状态变化或完成时，通过 `MessageBus` 向对应的 Channel 发送异步系统消息，从而唤醒主 Agent 或通知用户。

## 二、Agent 协同的“幻觉陷阱”：并行融合 (Merge) 是伪命题

**问题描述：**
原设计第 6 节提出了 `plan_codex` 和 `plan_claude` 独立并行规划，然后由主 Agent 执行 `plan_merge` 汇总。
由于大模型的特性，两份基于不同技术栈偏好生成的大型方案文档在合并时，极易造成上下文过载和逻辑冲突，最终产生充斥“幻觉”的“缝合怪”设计，极其不利于下游 Gemini 执行。

**改进方案：**
* **采用“提议-审查 (Proposer-Critic)” 的串行对抗模式**替代“并行-融合”模式。
* 步骤一：选定能力最强的模型输出初始方案（`plan_propose`）。
* 步骤二：将初稿作为上下文，交由另一模型进行缺陷审查、风险评估与补充（`plan_review`），从而在统一架构基调的前提下保证方案质量。

## 三、僵尸进程与崩溃恢复机制的缺失

**问题描述：**
如果 Neoclaw 进程在底层 `acpx` 执行期间因更新、OOM 或机器断电意外重启，原状态机和内存中的 Orchestrator 会丢失跟踪。底层的 `acpx` 将变成不受控的僵尸进程（Orphan Process）继续消耗资源，且结果无法回收。

**改进方案：**
* **加入跨重启的持久化协调机制（Reconciliation）。** 
* 在启动底层的系统进程时，将对应的 `PID` 和 `Session Identifier` 写入磁盘日志。
* Neoclaw 启动加载时，扫描持久化存储中的未完成 Runs，并校验对应的 PID 存活状态；具备继续追踪条件的重新 Attach，无主进程则明确标记 Failed 或杀掉残留重试。

## 四、上下文暴增与工件隔离机制

**问题描述：**
原设计笼统地提到“读取 implementation-spec”。在真实的复杂任务中，前置步骤（规划、测试、运行）会产生大量冗余日志和中间文件。如果将整个工作空间作为上下文传递，会瞬间耗尽后续执行节点（如 Gemini）的 Token 窗口，导致严重降智。

**改进方案：**
* **引入严格的“工件交接契约（Artifact Hand-off Contract）”。**
* 编排器在推动 DAG 运转时，上游节点必须输出明确的 `{step_id}.outputs.json`，声明哪些文件是“交接给下游的核心上下文”，哪些是“仅供人类审计的追踪日志”。
* 下游启动时，仅挂载（或在 Prompt 中注入）核心交接上下文，实行严格的信息阻断，保护模型的注意力机制。

## 五、人机共驾（HITL）与断点挂起（Suspend）状态

**问题描述：**
Codex 提到验证失败时进行“自动修复”。但现实中，复杂的依赖冲突或底层错误往往会让 Agent 陷入“改代码-报错-乱改代码”的死亡循环，消耗大量 Token 并彻底破坏代码库。单纯的 Cancel 机制又过于粗暴。

**改进方案：**
* **引入工作流“挂起（Suspend）”状态。**
* 当错误重试达到阈值，或遇到无法自动解决的严重冲突时，工作流应暂停并转入 `suspended_waiting_for_user` 状态。
* 通过 UI/消息提示用户介入（如手动修复 `/workspace/src` 的特定语法错误），用户修复完成后点击“继续（Resume）”，将人类智慧无缝嵌入到 Agent 编排的工作流中。
